# Образовательный слой (RU) — «под капотом» + «Повторите сами»

> Учебный слой к сайту. Frontend позже вплетёт его как раскрывающиеся callout-ы
> «Под капотом» на существующих секциях и как отдельную секцию «Как это устроено /
> Повторите сами». Это вторая половина dual mandate: сайт должен быть не только
> увлекательным, но и **по-настоящему обучать тому, как работать с Claude Code** —
> так, чтобы зритель мог это повторить.
>
> **Источник истины — точность важна, это туториал:** `docs/agent-teams.md` и то, что
> реально произошло в этой сборке. Если факт не проверяется по `docs/agent-teams.md`
> или нашему реальному прогону — мы его не пишем. Тон: русский, ёмкий, честный.
> Короткие учебные «самородки», не дамп мануала. Технические термины — как есть.

## A. «Под капотом» — по одному самородку на каждую секцию главной

Frontend вешает это как раскрывающийся блок «Под капотом ↓» под соответствующей секцией.

### A1. Hero
- **Под капотом:** `То, что вы читаете, собрали семь отдельных экземпляров Claude Code, а
  не один. Одна сессия — «лид» — раздаёт работу, остальные шесть работают каждый в своём
  контекстном окне и переписываются между собой. Это и есть «команда агентов».`
- **Повторите сами:** `Запустите Claude Code и опишите задачу вместе со структурой команды
  («создай команду: один на UI, один на API, один на тесты») — он предложит состав, вы
  подтвердите.`

### A2. «Как работают команды агентов в Claude Code» (`#teams`)
- **Под капотом:** `Ключевое отличие от субагентов: субагент живёт внутри одной сессии и
  только возвращает результат вызвавшему — между собой субагенты не общаются. Тиммейты в
  команде — полноценные сессии: общий список задач (захват защищён блокировкой файла) и
  почтовый ящик (SendMessage), через который они пишут друг другу напрямую.`
- **Повторите сами:** `Включите экспериментальный флаг в settings.json:
  `{"env": {"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"}}` и проверьте версию — нужна
  Claude Code v2.1.32+ (`claude --version`).`

### A3. «Наша структура» (две волны, барьеры, idle-during-QA) (`#how`)
- **Под капотом:** `Барьер — это не фича Claude Code, а дисциплина, которую держит лид: он
  не раздаёт следующую фазу, пока не выполнено условие (страница рендерится И API отвечает).
  Правило idle-during-QA означает, что во время теста лид не назначает разработчикам задач,
  а они сами ничего не разбирают из общего списка — иначе код под тестом изменится, и
  вердикт QA станет бессмысленным.`
- **Повторите сами:** `Социально — командуйте лиду «дождись завершения тиммейтов, прежде
  чем двигаться дальше». Жёстко — повесьте hook `TaskCompleted` (exit code 2 отменяет
  завершение задачи и шлёт фидбэк) или `TeammateIdle` (exit code 2 не даёт тиммейту уйти в
  простой и возвращает его к работе).`

### A4. «Скиллы, MCP и плагины» (`#tools`)
- **Под капотом:** `Скилл — это переиспользуемая инструкция «как делать X»: научил Claude
  однажды — и дальше он делает так каждый раз. MCP — это протокол, через который Claude
  подключается к внешним системам (Figma, браузер, генератор компонентов). Плагин просто
  упаковывает скиллы + агентов + MCP-серверы вместе, чтобы делиться ими одним пакетом.`
- **Повторите сами:** `Вызовите скилл по имени в сессии; подключите MCP-сервер в настройках;
  поставьте плагин — и его скиллы/агенты/MCP станут доступны всей команде.`

### A5. «Ошибки, которые мы сделали» (`#mistakes`)
- **Под капотом:** `Каждая ошибка превратилась в правило, записанное в
  `knowledge/course-corrections/`. Это и есть «непрерывное улучшение»: PM фиксирует урок
  человека или повторяющийся затык, лид применяет его перед следующей фазой. Правила
  «лид не кодит», «QA только тестирует», «idle-during-QA» родились именно так.`
- **Повторите сами:** `Заведите папку для course-corrections; после каждого факапа пишите
  «триггер → коррекция → почему» и читайте её в начале фазы. Можно закрепить hook-ами.`

### A6. Тизер команды / `/agents`
- **Под капотом:** `Каждая роль — это файл-определение `.claude/agents/*.md`. Из него
  тиммейт наследует `tools` (allowlist инструментов) и `model` (у нас у всех — Opus), а тело
  файла дописывается в системный промпт. Важная тонкость: поля `skills` и `mcpServers` из
  этого frontmatter тиммейту НЕ применяются — скиллы и MCP он берёт из project/user-настроек,
  как обычная сессия. Инструменты команды (SendMessage, задачи) доступны всегда, даже если
  `tools` всё урезал.`
- **Повторите сами:** `Создайте `.claude/agents/reviewer.md` с `tools` и `model`, затем
  спавните тиммейта по этому типу: «создай тиммейта типа reviewer для аудита auth-модуля».`

### A7. Сигнатурные фичи (тред / хронология / фильм)
- **Под капотом (тред):** `Реплики в треде — это и есть реальный механизм SendMessage:
  тиммейты и лид обменивались такими сообщениями, они доставляются автоматически (без
  поллинга), а простаивающий тиммейт сам уведомляет лида.`
- **Под капотом (фильм):** `Фильм собран Хроникёром на Remotion и встроен в сайт через
  @remotion/player — отдельный тиммейт с собственным набором инструментов, чья «партия» —
  только видео-композиция.`
- **Повторите сами:** `Дайте одному тиммейту узкую зону файлов (например, только видео или
  только API) — так роли не конфликтуют и строят параллельно.`

## B. Глубокие карточки для секции «Как это устроено» (учебные нуггеты)

Frontend может вынести это в отдельную секцию/таб. Каждая карточка: что это → как мы
использовали → как повторить. Всё проверяется по `docs/agent-teams.md` и нашему прогону.

### B1. Команды агентов (agent teams)
- **Что это:** `Координация нескольких экземпляров Claude Code. Лид = главная сессия
  (раздаёт задачи, синтезирует). Тиммейты = отдельные сессии, каждая в своём контексте,
  общаются напрямую.`
- **Механика (точно):**
  - `Общий список задач с зависимостями: завершение задачи разблокирует зависимые; захват
    задачи использует блокировку файла против гонок; состояния — pending / in progress / completed.`
  - `Почтовый ящик: SendMessage между агентами; доставка автоматическая, без поллинга.`
  - `Хранилище управляется автоматически и существует только пока команда активна
    (config + список задач); руками его не редактируют.`
  - `Лимиты: одна команда за раз, вложенных команд нет, лид фиксирован на время жизни
    команды, права задаются при спавне.`
- **Как мы использовали:** `Лид раздавал задачи фронтенду/бэкенду, держал барьеры; тиммейты
  переписывались (см. секцию-тред) и разбирали задачи из общего списка.`
- **Повторите сами:** `(1) Claude Code v2.1.32+. (2) В settings.json:
  `{"env":{"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS":"1"}}`. (3) Опишите задачу + состав
  команды текстом — Claude создаст команду и заспавнит тиммейтов.`

### B2. Роли как определения субагентов
- **Что это:** `Переиспользуемое описание роли в `.claude/agents/*.md`. Работает и как
  субагент, и как тиммейт.`
- **Механика (точно):** `Наследуются `tools` (allowlist) и `model`; тело файла дописывается
  в системный промпт (не заменяет его). **`skills` и `mcpServers` из frontmatter тиммейту
  НЕ применяются** — он грузит скиллы/MCP из project/user-настроек. Командные инструменты
  (SendMessage, задачи) доступны всегда.`
- **Как мы использовали:** `Семь ролей (Маэстро, Либреттист, Сценограф, Машинист, Камертон,
  Рецензент, Хроникёр), у всех `model: opus`; у QA и Reviewer — урезанные `tools` под их
  «только тестировать» / «только читать».`
- **Повторите сами:** `Опишите роль один раз с нужным `tools`+`model`, ссылайтесь на неё
  при спавне тиммейта или субагента.`

### B3. Барьеры, idle-during-QA и гейты качества
- **Что это:** `Дисциплина передачи работы между фазами, которую держит лид.`
- **Механика (точно):** `Сам Claude Code даёт для этого hook-и: `TaskCompleted` (exit 2
  отменяет завершение задачи + шлёт фидбэк), `TeammateIdle` (exit 2 не пускает тиммейта в
  простой), `TaskCreated` (exit 2 запрещает создание задачи). Барьер и idle-during-QA можно
  держать социально (через лида) и/или закрепить этими hook-ами.`
- **Как мы использовали:** `Барьер «фича готова» → гейт QA (devs idle) → гейт Reviewer
  только после PASS. Никто не проверяет собственную работу.`
- **Повторите сами:** `Скажите лиду «не начинай ревью, пока QA не вернёт PASS», или повесьте
  `TaskCompleted`-hook, который не даёт закрыть задачу без зелёного QA.`

### B4. Скиллы (skills)
- **Что это:** `Инструкция «как делать X», которую Claude применяет одинаково каждый раз.`
- **Реальные скиллы и что каждый сделал:**
  - `brainstorming — прояснил намерение и требования до кода.`
  - `writing-plans — превратил спецификацию в пошаговый план с волнами.`
  - `ui-ux-pro-max — выбор дизайн-направления: палитры, шрифтовые пары, тип продукта.`
  - `frontend-design — отчётливый «студийный» UI без шаблонной ИИ-эстетики.`
  - `frontend-patterns — паттерны React/Next: состояние, производительность.`
  - `backend-patterns — дизайн API, работа с БД, серверные best practices.`
  - `security-review — чек-лист безопасности: ввод, секреты, утечки ошибок.`
  - `design-taste-frontend / high-end-visual-design / gpt-taste — независимая критика вкуса.`
  - `brandkit — токены бренда (цвет, типографика, тон), записанные один раз.`
  - `remotion / motion-graphics / svg-animation — раскадровка и анимация фильма Хроникёра.`
- **Повторите сами:** `Вызовите скилл по имени в сессии — например, попросите «спланируй
  через writing-plans» или «оцени вкус через ui-ux-pro-max».`

### B5. MCP (Model Context Protocol)
- **Что это:** `Протокол, через который Claude подключается к внешним системам и инструментам.`
- **Реальные серверы и что каждый дал:**
  - `Figma — мост дизайн ↔ код: токены и контекст макета в обе стороны.`
  - `21st.dev magic — генерация готовых UI-компонентов, из которых собиралась страница.`
  - `Chrome / claude-in-chrome (+ computer use) — QA водит настоящий браузер: клики,
    чтение страницы и консоли, скриншоты/gif.`
  - `Playwright — браузерная автоматизация для воспроизводимых E2E-сценариев.`
- **Честно (реальный затык):** `На первом дизайн-подходе через Figma мы упёрлись в
  per-seat rate limit Figma MCP и встали. Урок: у внешних MCP есть лимиты — это часть
  планирования, а не сюрприз.`
- **Повторите сами:** `Подключите MCP-сервер в настройках Claude Code; для QA дайте тиммейту
  инструменты `mcp__claude-in-chrome__*`, чтобы он тестировал живой UI, а не читал код.`

### B6. Плагины (plugins)
- **Что это:** `Пакет, упаковывающий скиллы + субагентов + MCP-серверы вместе, чтобы делиться
  одним набором.`
- **Реальные плагины:** `ui-ux-pro-max (дизайн-интеллект), everything-claude-code (агенты,
  ревью, паттерны, TDD), figma (официальный мост к Figma), playwright (E2E),
  claude-in-chrome (управление Chrome).`
- **Тонкость:** `Плагин ставит скиллы и MCP в project/user-настройки — именно оттуда их и
  берут тиммейты (помните: frontmatter `skills`/`mcpServers` роли им не применяется).`
- **Повторите сами:** `Поставьте плагин — и его скиллы/агенты/MCP станут доступны всей
  команде из общих настроек.`

## C. Предлагаемая секция «Как это устроено / Повторите сами» — outline

Frontend-у на сборку (отдельный раздел или таб, можно якорь `#under-the-hood`):

- **Надзаголовок:** `Под капотом`
- **Заголовок (H2):** `Как это устроено — и как повторить.`
- **Лид-абзац:** `Этот сайт не только рассказывает историю, но и учит механике. Всё ниже —
  реальные приёмы Claude Code, которыми мы пользовались. Каждый блок заканчивается одним-двумя
  конкретными шагами, чтобы вы могли сделать так же.`
- **Карточки (из раздела B, в этом порядке):**
  1. `Команды агентов` (B1)
  2. `Роли как определения субагентов` (B2)
  3. `Барьеры, idle-during-QA и гейты` (B3)
  4. `Скиллы` (B4)
  5. `MCP` (B5)
  6. `Плагины` (B6)
- **Формат каждой карточки:** `Что это → Как мы использовали → Повторите сами (1–2 шага).`
- **Блок «Быстрый старт» (внизу секции):**
  1. `Claude Code v2.1.32+ — проверьте `claude --version`.`
  2. `Включите команды агентов: `{"env":{"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS":"1"}}` в settings.json.`
  3. `Опишите задачу и состав команды текстом — Claude создаст её и заспавнит тиммейтов.`
  4. `Дайте каждой роли свою зону файлов (UI / API / тесты / видео) — параллельно, без конфликтов.`
  5. `Держите барьеры: «не начинай ревью, пока QA не вернёт PASS» — или закрепите hook-ами.`
- **Завершающая строка:** `Всё, что вы видели на этой странице, собрано именно так.`

## Заметки для Frontend (не выводить)

- **Accuracy first:** каждый факт здесь проверен по `docs/agent-teams.md` или нашему прогону.
  Не добавлять непроверяемых утверждений (версии, флаги, имена hook-ов — не менять).
- **Точные имена (не перефразировать):** `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`,
  `v2.1.32+`, hook-и `TaskCompleted` / `TeammateIdle` / `TaskCreated` (exit code 2),
  `.claude/agents/*.md`, `SendMessage`, `mcp__claude-in-chrome__*`, `@remotion/player`.
- **Формат:** короткие самородки, раскрывающиеся «Под капотом ↓»; не стена текста.
- **Связка с дизайном:** «Под капотом» — это второй, опциональный слой поверх увлекательного;
  по умолчанию свёрнут, раскрывается по клику. Excitement остаётся первичным.
